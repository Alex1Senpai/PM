# Task workflow — конвенции + автоматизация

Этот документ — операционный свод правил для Claude-агента разработчика. Когда Alex или Timur говорит «Начни работу над T-001» или «Заверши T-001», Claude следует этому workflow строго.

Если конвенция здесь конфликтует с практикой команды — это дефект документа, а не повод нарушить конвенцию. Сообщи Bekhruz, обновим.

---

## Именование бранчей

- Формат: `task/T-NNN-короткий-slug`
- Slug: только нижний регистр, дефисы (не подчёркивания), максимум 30 символов после `T-NNN-`
- Пример: `task/T-001-flutter-auth-otp` ✓
- Анти-примеры: `task/T_001_Auth_OTP` ✗, `task/T-001_Flutter_Auth_And_OTP_Biometric_Extra_Long` ✗

## Формат коммитов

- Формат: `T-NNN: imperative description` (первая строка до 72 символов)
- Несколько задач (редко): `T-NNN T-MMM: общее изменение`
- Императивная форма: `Add`, `Fix`, `Refactor` — не `Added`, `Fixed`, `Refactored`
- Детальное описание после пустой строки — опционально, без ограничения длины, можно на русском
- Пример: `T-001: scaffold flutter project + auth flow`

## Заголовки PR и описания

- Заголовок PR: тот же формат что коммит — `T-NNN: description`
- Описание PR: автозаполняется из `.github/PULL_REQUEST_TEMPLATE.md` — заполни все секции
- Ссылка на PM-задачу: `Closes: PM/Funnel/.../T-NNN-*.md`

## Начать задачу — «Начни работу над T-NNN»

1. Прочитай файл задачи в PM. Проверь: `status: pending`, `commit_initial: ""`.
2. Подтверди с разработчиком: дата `target-done`, есть ли блокеры, понятен ли scope?
3. В код-репозитории создай бранч от main: `task/T-NNN-slug` (slug по правилам выше)
4. Пустой initial коммит: `git commit --allow-empty -m "T-NNN: start"`
5. Получи SHA: `git rev-parse HEAD`
6. Обнови frontmatter задачи в PM:
   - `status: pending` → `status: in_progress`
   - `commit_initial: <SHA>`
   - `code_repo: <URL репозитория>/tree/<branch>`
7. Закоммить изменение в PM: `T-NNN: start work, set commit_initial`
8. Push'и оба репозитория
9. Сообщи разработчику: «Бранч создан. Frontmatter обновлён. Можно писать код.»

## Во время работы

- Каждый коммит начинается с `T-NNN:`
- Ссылайся на Decisions в теле коммита, когда применимо: `Per D-006, use Dart on server.`
- Если всплыло новое архитектурное решение — **пауза**. Предложи новый файл `D-NNN-*.md` в PM, получи одобрение Bekhruz, и только потом продолжай.

## Закрыть задачу — «Заверши T-NNN»

1. Получи финальный SHA коммита на бранче
2. Вызови `_Skills/complete-task.md` для генерации раздела «Завершение»
3. Покажи «Завершение» разработчику, дай отредактировать при необходимости
4. Обнови файл задачи в PM:
   - `status: in_progress` → `done`
   - `commit_final: <SHA>`
   - Вставь раздел «Завершение»
   - Допиши запись в «История изменений»
5. Открой PR в код-репозитории с заполненным template'ом
6. Открой PM PR с обновлениями файла задачи
7. Пришли разработчику ссылки на оба PR

## Жёсткие правила — нарушать нельзя

- Никогда не пуш прямо в `main` (всегда через PR)
- Никогда force-push в shared-бранч
- Никогда не редактируй `.git/` или `.gitignore` без явной просьбы
- Никогда не коммить секреты (`.env`, `*.key`, `*.pem`, `*.p12`, `*.keystore` — они в .gitignore)
- Никогда не пропускай префикс `T-NNN:`
- Если scope непонятен — **спроси** разработчика или Bekhruz, прежде чем действовать

---

## Проверка конвенций — «Проверь конвенции»

Когда вызвано, Claude сканирует недавнюю активность и докладывает:
- Бранчи, не соответствующие формату
- Коммиты без префикса `T-NNN:`
- Задачи `in_progress` без активности по коммитам более 5 дней
- Задачи `done` с незаполненным `commit_final`
- PR без заполненных секций template'а

## Что делать, если задачи нет (no-task workflow)

Если разработчик хочет сделать что-то, на что нет файла задачи в PM (нашёл баг, хочет рефактор, нужна правка документации):

1. **Не делай работу сразу.** Сначала создай задачу.
2. Предложи новый файл `T-NNN-*.md` в `Funnel/<стадия>/Tasks/` по шаблону `_Templates/Task.md`
3. Заполни Контекст, Что нужно сделать, Definition of Done — даже грубо, на 70%
4. Покажи Bekhruz, получи одобрение
5. После одобрения — начинай задачу по обычному workflow выше

**Исключение для тривиальных правок:** опечатка в README, очевидный обновлённый комментарий — можно сделать через `task/T-NNN` где NNN — следующий свободный номер, но всё равно создать задачу пост-фактум для аудита.

## Требования к PR перед мержом (минимум для approve)

| Требование | Применимо когда |
|---|---|
| Все пункты Definition of Done отмечены | Всегда |
| PR описание заполнено по template | Всегда |
| Префикс `T-NNN:` в каждом коммите | Всегда |
| Тесты добавлены или обновлены | Когда меняется бизнес-логика, новая функция, или фикс бага |
| Прохождение CI | Когда CI настроен (после первых тестов в репо) |
| Скриншот/staging URL | Когда изменения видимы в UI |
| Архитектурный сюрприз задокументирован | Когда было принято нестандартное решение |
| Связанные Decisions актуализированы | Когда работа меняет/реализует решение |

Bekhruz решает финальный merge. Эти требования — рекомендация Claude, а не override Bekhruz.
