# Complete-task — генерация «Завершения» из git-истории

Этот навык создаёт раздел «Завершение» в файле задачи. Разработчик вызывает его (или Claude автоматически вызывает в рамках `task-workflow.md`), когда задача готова к закрытию. Результат проверяется разработчиком, при необходимости редактируется, и вставляется в файл задачи.

## Что нужно навыку на вход

Разработчик (или вызывающий Claude) предоставляет:
1. **ID задачи** — например, `T-001`
2. **Путь к файлу задачи** — например, `Funnel/3-Parent-App/Tasks/T-001-flutter-scaffolding-auth.md`
3. **URL кодового репозитория + бранч**
4. **`commit_initial` SHA** — из frontmatter файла задачи (установлен в начале работы)
5. **`commit_final` SHA** — последний коммит на бранче
6. **Вывод git log:**
   ```bash
   git -C <path_to_code_repo> log --no-merges --pretty=format:"%h %s%n%b%n---" <commit_initial>..<commit_final>
   git -C <path_to_code_repo> diff --stat <commit_initial>..<commit_final>
   ```
   Если SHA коммитов потерялись (rebase, force-push), используй fallback:
   ```bash
   git -C <path_to_code_repo> log --no-merges --pretty=format:"%h %s%n%b%n---" --grep="T-NNN"
   ```

## Что делает навык

1. **Читает файл задачи** по предоставленному пути
2. **Читает раздел «Что нужно сделать»** — это шаги, которые должны были произойти
3. **Читает раздел «Определение готовности»** — это проверки, которые доказывают «готово»
4. **Читает git log + diff stats** от разработчика
5. **Кросс-ссылка:** маппит каждый коммит на шаг или пункт DoD. Отмечает пункты, у которых нет соответствующего коммита (могли быть сознательно пропущены или не сделаны).
6. **Генерирует раздел «Завершение»** в этой структуре:

```markdown
## Завершение

**Что запущено:**
- {{1 строка-итог по каждой крупной части, сопоставленной с коммитами}}

**Где это лежит:**
- **Код:** {{URL репозитория + бранч + финальный SHA коммита}}
- **Deploy / staging URL:** {{если разработчик упоминал в коммитах}}
- **Связанные файлы (по diff):** {{ключевые пути из --stat}}

**Что НЕ сделано из изначального плана (если что-то не сделано):**
- {{пункты из «Что нужно сделать», у которых нет коммита, ИЛИ пометка «всё сделано по плану»}}

**Сюрпризы и решения в процессе:**
- {{всё в commit messages, что указывает на неочевидное design-решение, рефакторинг или workaround}}

**Что нужно знать следующему разработчику:**
- {{actionable handoff notes: вещи, о которых должен знать следующий человек (env переменная X нужна, миграция Y должна запуститься первой)}}
- {{если ничего особенного — «нет специфических заметок»}}

**Метрики:**
- Коммитов: {{count}}
- Файлов изменено: {{из --stat}}
- Строк +/-: {{из --stat}}
- Период работы: {{от даты первого коммита до даты последнего}}
```

## Правила качества вывода

- **Не выдумывай.** Если в коммитах нет упоминания факта — не пиши, что это сделано.
- **Будь конкретен.** «Реализован экран авторизации» — плохо. «Реализован экран ввода номера телефона с валидацией +998 и переходом на OTP-экран по нажатию Получить код» — хорошо.
- **Цитируй пути файлов** из diff stat, когда это помогает контексту: «Логика валидации — `lib/auth/phone_validator.dart`».
- **Если есть тесты — упомяни их явно:** «Добавлено 7 unit-тестов на AuthService».
- **Если в коммитах есть TODO, FIXME, или XXX** — упомяни в «Что не сделано».
- **Сюрпризы — самая важная секция.** Если в commit message написано «решил использовать X вместо Y потому что Z» — это сигнал, что есть архитектурное решение, которое нужно зафиксировать. Возможно, должно стать Decision-файлом.

## Когда Claude должен пушбэкнуть и не генерировать

- Если `commit_initial` или `commit_final` отсутствуют, и `--grep` тоже ничего не находит: ответь «Не нашёл коммитов для этой задачи. Возможно, ID задачи не использовался в commit messages. Проверь git log вручную или дай прямой список SHA».
- Если в git log нет осмысленных коммитов — только «WIP», «fix», «update» — ответь «Сообщения коммитов слишком неинформативны для генерации завершения. Допиши описания через `git commit --amend` или напиши завершение вручную».
- Если ни одно из условий DoD не покрыто в diff — ответь «По коммитам не видно, что задача выполнена. Уверен, что это правильная пара commit_initial / commit_final?»

## Discovery-задачи (без кода)

Если задача — это discovery (например, T-002 SMS-провайдер, T-003 1C API, T-004 aggregator):
- `code_repo: n/a` или `code_repo: ""` в frontmatter
- Нет коммитов в код-репозиторий
- «Завершение» генерируется из заполненных секций самого файла задачи (например, «Сравнительная таблица провайдеров», «Рекомендация»)

Структура для discovery «Завершения»:
```markdown
## Завершение

**Что выяснено:**
- {{краткий пересказ ключевых находок}}

**Рекомендация:**
- {{из секции «Рекомендация» файла задачи}}

**Открытые вопросы:**
- {{если что-то не получилось выяснить}}

**Следующие шаги:**
- {{что нужно сделать по итогам discovery}}
```

## Использование

Самый простой способ — открыть Claude (этого же или другого) и вставить запрос:

```
Сгенерируй раздел «Завершение» для задачи T-NNN, используя _Skills/complete-task.md.

Файл задачи: <путь к Task.md>

Кодовый репозиторий: <URL + бранч>
commit_initial: <SHA>
commit_final: <SHA>

Git log:
<вставь сюда вывод git log>

Diff stat:
<вставь сюда вывод git diff --stat>
```

Claude вернёт готовый раздел «Завершение» — прочитай, поправь, вставь в файл задачи. Затем коммить и открывай PM PR.
