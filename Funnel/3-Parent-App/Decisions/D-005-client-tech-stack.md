# D-005 — Технологический стек клиента Parent App

## Контекст
Parent App нужно запустить на iOS и Android к 15 августа 2026. Команда — Алекс (middle dev) + Тимур (intern). Срок жёсткий, объём v1 — 5–6 поверхностей.

Решение о клиентском стеке определяет скорость разработки, доступ к нативным API (биометрия, push, камера), и стоимость будущих доработок.

## Рассматриваемые варианты

### Вариант A — Нативный (Swift для iOS + Kotlin для Android)
- **Описание:** Две отдельные кодовые базы. Каждая фича пишется дважды.
- **Плюсы:** Лучший UX, прямой доступ к API платформы
- **Минусы:** Двойная работа. С одним middle dev + интерном в 12 недель — нереалистично. К 15 августа было бы 2–3 поверхности максимум.
- **Отклонён:** не вписывается в дедлайн с текущей командой.

### Вариант B — React Native
- **Описание:** Один код, JavaScript/TypeScript. Скомпилировано через мост к нативным компонентам.
- **Плюсы:** Большой ecosystem, много готовых решений, большой talent pool для будущих наймов в Ташкенте
- **Минусы:** Алекс не знает JS/TS — ему пришлось бы изучать новый язык и фреймворк параллельно с разработкой. Потеря недель на onboarding в фреймворк.
- **Отклонён:** не подходит под backгроund Алекса.

### Вариант C — Flutter / Dart (принят)
- **Описание:** Один код, Dart. Компилируется в нативный код для iOS и Android. Material / Cupertino design systems встроены.
- **Плюсы:**
  - **Алекс уже Flutter/Dart разработчик** — нулевое время на изучение
  - Один код для обеих платформ
  - Хорошая производительность (нативная компиляция)
  - Готовые библиотеки для биометрии (`local_auth`), push (`firebase_messaging`), камеры, и т.д.
  - Хороший toolchain (Flutter CLI, hot reload, отладка)
- **Минусы:**
  - Меньший talent pool в Ташкенте чем у React Native — будущие найма будут сложнее
  - Некоторые интеграции (Hikvision, специфичные узбекские SDK) могут потребовать platform channels — пишется на Swift / Kotlin
  - Меньший ecosystem чем RN, но достаточный для нашего scope
- **Стоимость:** Минимальная относительно текущей команды. Будущая цена: при найме новых разработчиков в 2027+ Flutter-разработчиков в Ташкенте может быть меньше, чем React Native.

### Вариант D — PWA (Progressive Web App)
- **Описание:** Веб-приложение, доступное через браузер с возможностью «Установить на главный экран».
- **Плюсы:** Самая простая дистрибуция, нет необходимости в магазинах приложений
- **Минусы:**
  - На iOS PWA имеет слабую поддержку push-уведомлений
  - Биометрический вход на iOS PWA ограничен
  - Родители ищут приложения в App Store / Google Play — отсутствие там подрывает доверие, особенно для приложения с финансовыми данными
  - Семьи привычны к нативным банковским приложениям — PWA «чувствуется» менее серьёзно
- **Отклонён:** не подходит для финансового приложения с push и биометрией.

## Решение
**Вариант C — Flutter / Dart.** Один Flutter-проект, билдится в нативные iOS и Android приложения. Используем стандартные пакеты Flutter ecosystem для биометрии, push, REST API, и платёжных интеграций.

## Почему
- **Совпадение с background Алекса.** Это решающий фактор. Любой другой стек требует, чтобы Алекс учился во время разработки. Потеря недель.
- **Один кодbase, два релиза.** Соответствует scope в 12 недель.
- **Достаточная глубина ecosystem.** Все нужные нам интеграции (биометрия, push, камера, REST) имеют зрелые Flutter-пакеты.

## Последствия
- **Что мы теперь обязаны сделать:**
  - Настроить Flutter-проект (Alex)
  - Выбрать architecture pattern внутри Flutter (Provider? Bloc? Riverpod?) — это его архитектурное решение, не моё
  - Настроить CI/CD для билдов и публикации в App Store / Google Play
  - При появлении интеграций без готовых Flutter-пакетов — писать platform channels (мост к нативному коду)
- **Что мы не можем легко поменять:**
  - Переписать на React Native или нативный позже — это полная переразработка
- **Что это открывает:**
  - Возможность переиспользовать Flutter-код для будущих surfaces (Teacher App, Admin Dashboard) если решим, что они тоже мобильные

## Открытые подвопросы (Alex решает в spec)

- Какой backend — Node.js, Python, Dart на сервере (Shelf), Go? **Это отдельное решение D-006, не входит сюда.**
- Какой минимум поддерживаемых OS — iOS 15? Android 9?
- Использовать ли Firebase as Backend (FBaaS) или поднимать свой backend

## Влияние на интеграции и LMS
- Не integration-impacting. Стек клиента не влияет на контракты с источниками данных. API между клиентом и любым backend / источником остаются REST/JSON.
- В 2027–28 при появлении LMS Flutter-клиент адаптируется к новым API без переписывания.

## Будущие риски
- Если в 2027+ нужно нанять второго мобильного разработчика, найти Flutter в Ташкенте сложнее, чем React Native. Принимаем сознательно — Алекс существует сейчас, гипотетический найм — потом.
- Если в 2027+ решим переходить на нативный или React Native — это полная переразработка клиента.

## Пересмотреть когда
- Если у Алекса возникают серьёзные блокеры в Flutter, которые ставят дедлайн под угрозу
- При найме новых разработчиков в команду — если они не Flutter, может потребоваться переоценка
- В 2027–28 при работе над LMS и Teacher App — переоценить, остаётся ли Flutter правильным выбором
