# D-007 — Модель синхронизации между Parent App backend и 1C

## Контекст
После D-001 (Variant C — backend как кеш + 1C как источник истины) необходимо определить, как данные перемещаются между нашим backend и 1C.

Это решение задаёт **архитектурную рамку**. Конкретные технические детали (частота опроса, формат API, библиотеки) — выбор Алекса после discovery 1C-стороны.

Два направления потока данных, и они **различаются по природе**:

1. **Платежи** — поступают через Click/PayMe webhook'и. Это HOT-данные: родитель ждёт подтверждения через секунды.
2. **Балансы и корректировки** — управляются бухгалтером в 1C. Это COLD-данные: задержка в десятки минут приемлема.

Эти потоки требуют разных моделей синхронизации. Использовать одну модель для обоих — компромисс хуже обоих по отдельности.

## Рассматриваемые варианты

### Вариант A — Real-time API (Parent App backend → 1C на каждый запрос)
- Отклонён ранее в D-001. Каждый запрос баланса бьёт в 1C → 1C-API медленный, единственная точка отказа, нет смысла в кеше.

### Вариант B — Push от 1C к нашему backend (1C вызывает наш webhook при изменениях)
- **Минусы:** Требует кастомизации 1C на стороне клиента. Многие 1C-конфигурации не умеют push'ить. Зависимость от 1C-разработчика.
- **Не выбран:** ненадёжно как единственный механизм.

### Вариант C — Pull от backend к 1C (polling/опрос)
- Наш backend периодически запрашивает 1C
- **Под-варианты по частоте:** 1 минута, 15 минут, 1 час, ночной batch
- **Плюсы:** контроль на нашей стороне, 1C не меняется
- **Минусы:** задержка в N минут, нагрузка на 1C-API
- **Реалистично** для COLD-данных (баланс, корректировки)

### Вариант D — Event-driven через очередь
- 1C публикует события в очередь (RabbitMQ, Kafka), backend подписывается
- **Минусы:** требует publishing с 1C-стороны, дополнительная инфраструктура (очередь, broker), оверхед для нашего масштаба
- **Отклонён:** слишком сложно для 1500 пользователей.

### Вариант E — Ручной export/import (CSV из 1C)
- **Минусы:** ручной, ошибочный, не масштабируется на 1500+ платежей в месяц
- **Отклонён:** не подходит для production.

### Вариант F — Гибрид (принят)
- Платежи: webhook от Click/PayMe → наш backend → асинхронный push в 1C
- Баланс и корректировки: наш backend pull'ит из 1C по расписанию
- Reconciliation: при расхождении 1C всегда выигрывает

## Решение
**Вариант F — Гибридная модель синхронизации.**

### Поток 1: Платежи (HOT)
1. Родитель оплачивает через Click или PayMe
2. Click/PayMe отправляет webhook → **наш backend**
3. Backend записывает платёж в свою базу (immediate: parent видит подтверждение)
4. Backend асинхронно push'ит запись о платеже в 1C (через 1C REST API, OData, или другой механизм — конкретику определит Алекс после discovery)
5. При неудаче push в 1C: повторные попытки с exponential backoff. Если 1C недоступен дольше N часов — алерт администратору.
6. Все push-операции **идемпотентны**: каждый платёж имеет уникальный transaction ID, повторная попытка не создаёт дубликат.

### Поток 2: Баланс и корректировки (COLD)
1. Бухгалтер делает изменения в 1C (начисления, ручные корректировки, возвраты)
2. Наш backend **pull'ит** из 1C по расписанию: **каждые 15 минут до 1 часа** (точную частоту выбирает Алекс после profiling 1C-API)
3. Backend сравнивает полученные данные со своим кешем
4. При расхождении: **1C всегда выигрывает** — backend обновляет свой кеш до значения 1C
5. При нагрузке: можно временно увеличить интервал опроса; ничего критичного не ломается, только увеличивается задержка отображения

### Reconciliation (выверка)
- **Правило:** В случае расхождения backend ↔ 1C — приоритет всегда у 1C
- **Логирование:** Каждое расхождение пишется в `reconciliation_log` для разбора
- **Notifications:** Если за день >5 расхождений (порог TBD) — алерт администратору
- **Daily reconciliation job:** Ночной cron в 3:00 — полная сверка всех активных балансов между backend и 1C, отчёт администратору утром

## Почему
- **Платежи нужны мгновенно.** Родитель ждёт подтверждения. Webhook'и идут в наш backend → секунды до подтверждения. Затем асинхронный push в 1C не блокирует UX.
- **Баланс может опаздывать на 15–60 минут.** Бухгалтер делает корректировки не каждую секунду. Pull-опрос приемлем.
- **1C всегда источник истины при конфликтах.** Это закрывает юридический вопрос — даже если в Parent App что-то отображается неверно, при возникновении спора смотрим в 1C.
- **Идемпотентность push'ей** защищает от дубликатов при ретраях.

## Что определяет Алекс после discovery 1C-API (D-008 уровень детали)
- Каким образом push'ить платежи в 1C: REST? OData? 1C HTTP-сервис? Кастомный endpoint?
- Какой формат данных принимает 1C
- Какова реальная latency 1C-API → выбор частоты polling (15 мин vs 30 мин vs 1 час)
- Какие endpoints 1C предоставляет для чтения балансов
- Authentication: API token? Базовая авторизация? OAuth?
- **Действие:** Создать discovery-задачу T-NNN «Discovery: API 1C для Parent App backend». Назначить — обсудить (вероятно, Алекс + кто-то из 1C-стороны).

## Принимаемые риски
- **Латентность push'а платежей в 1C.** Если 1C недоступен — родитель видит платёж, но бухгалтер не видит. До восстановления связи — расхождение. Митигация: алерты + retry queue.
- **Задержка отображения корректировок.** Бухгалтер сделал корректировку в 1C → родитель видит её через 15–60 минут. Приемлемо для нашего домена.
- **Нагрузка на 1C-API.** Polling даёт нагрузку. Митигация: подобрать частоту по profiling.

## Последствия
- **Что мы теперь обязаны сделать:**
  - Создать задачу: Discovery 1C-API (что умеет, какой формат, какая авторизация)
  - В backend: реализовать idempotent push-механизм
  - В backend: реализовать polling-job + reconciliation logic
  - Настроить daily reconciliation cron + алерты
  - Логирование всех push/pull операций (для отладки и аудита)
- **Что мы не можем легко поменять:**
  - Базовая модель (push для платежей, pull для остального) — закладывается на v1, менять дорого
- **Что это открывает:**
  - Возможность добавить ещё одну сторону (Hikvision для проходных, Toddle для посещаемости) с теми же принципами

## Влияние на интеграции и LMS

- **2027–28 LMS:** Когда заменим Toddle на LMS — то же самое для student-данных (push от событий, pull для master-данных). Архитектурный шаблон переиспользуется.
