# D-009 — Прямая интеграция с Click + PayMe vs использование платёжного агрегатора

## Контекст
По решению D-003 Parent App принимает платежи в UZS через Click и PayMe (международные карты отклонены). Возникает вопрос **как** реализовать приём:

- **Прямая интеграция:** наш backend подключается к Click API и PayMe API отдельно. Два webhook-приёмника, два набора credentials, два набора edge cases.
- **Платёжный агрегатор:** одна интеграция с агрегатором (например, Rahmat), который скрывает за собой Click + PayMe + другие провайдеры. Один API, один webhook, один контракт.

Это решение влияет на скорость разработки, контроль над UX, и стоимость комиссии.

## Рассматриваемые варианты

### Вариант A — Прямая интеграция с Click + PayMe
- **Описание:** Наш backend пишет интеграции для Click и PayMe отдельно. Каждый провайдер — свой SDK/REST-клиент, свой webhook-эндпоинт, свои метаданные.
- **Плюсы:**
  - Прямые комиссии Click/PayMe (не платим агрегатору сверху)
  - Полный контроль над платёжным UX
  - Нет зависимости от третьей стороны
  - При проблемах общаемся напрямую с провайдером
- **Минусы:**
  - Две интеграции вместо одной — больше кода
  - Два webhook-приёмника со своей логикой
  - Два набора edge cases (разные форматы ответа, разные коды ошибок)
  - Каждая интеграция требует отдельной discovery, договора, подключения мерчанта
- **Стоимость:** Примерно 2 недели работы Алекса (по 1 неделе на провайдера). Комиссии — стандартные провайдерские.

### Вариант B — Агрегатор (Rahmat или аналог)
- **Описание:** Подключаемся к одному агрегатору. Он скрывает Click, PayMe, и возможно другие провайдеры за единым API. Родитель в UI видит выбор: «Click | PayMe | ...», но технически мы говорим только с агрегатором.
- **Плюсы:**
  - Одна интеграция вместо двух — экономия 50%+ времени разработки
  - Один webhook-эндпоинт
  - Единый формат ответа и ошибок
  - Один договор с агрегатором, не два
  - Легче добавлять новых провайдеров в будущем (включить через агрегатора)
- **Минусы:**
  - **Комиссия агрегатора сверху** провайдерской — выше итоговая стоимость каждой транзакции
  - **Зависимость от агрегатора** — если он лёг, лежим мы
  - **Меньше контроля над UX** — некоторые агрегаторы навязывают свой checkout-flow
  - **Дополнительная точка отказа** — лишний прыжок в платёжном пути
- **Стоимость:** Примерно 1 неделя работы Алекса (одна интеграция). Комиссия — провайдерская + надбавка агрегатора.

## Открытые вопросы (нужны для решения)
- [ ] Какова реальная надёжность Rahmat? Uptime, скорость, поддержка?
- [ ] Какая надбавка к комиссии? (Может быть от 0.1% до 1%+)
- [ ] Какие провайдеры он покрывает? Только Click + PayMe или больше?
- [ ] Какой checkout-flow он навязывает? Можем ли мы держать UX внутри Parent App?
- [ ] Какова процедура подключения? Сколько времени?
- [ ] Есть ли он у других школ / похожих B2C продуктов в Узбекистане?
- [ ] Какие альтернативные агрегаторы существуют (Octo и т.д.)?

## Моя рекомендация
Этот вопрос требует discovery до принятия решения. Я (Claude) на границе своей компетенции по специфике узбекских платёжных агрегаторов — данные, которые есть в открытых источниках, недостаточно детальны для уверенного выбора.

**Предлагаю создать новую discovery-задачу** (T-NNN) для Timur'а или Алекса: исследовать Rahmat и 2–3 альтернативных агрегатора, привезти данные по 7 открытым вопросам выше. Срок — 1 неделя. После этого закроем D-009.

**Дефолт, если discovery даст плохие результаты по агрегаторам:** Вариант A (прямая интеграция). Click и PayMe — стандарт, мы знаем что они работают.

**Дефолт, если discovery даст хорошие результаты:** Вариант B (агрегатор). Экономия недели разработки оправдывает небольшую надбавку к комиссии.

## Решение
*Пока не принято — `proposed`. Требует discovery (новая задача).*
