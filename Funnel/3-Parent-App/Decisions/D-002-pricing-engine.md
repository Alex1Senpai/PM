# D-002 — Где считаются скидки: семейная льгота и льгота раннего платежа

## Контекст
В Parent App нужно показывать корректный баланс с учётом двух типов скидок:

1. **Семейная льгота** (sibling benefit) — −5% за второго ребёнка, −10% за третьего и последующего. Применяется к ежемесячной оплате обучения. Зависит от состава семьи.
2. **Льгота раннего платежа** (early payment benefit) — −10% при оплате до 20 мая, −8% до 20 июня, −6% до 1 августа. Применяется только к программе Futurum (IB). Зависит от даты платежа.

Вопрос: **где живёт эта логика — в источнике баланса или в Parent App?**

Решение тесно связано с D-001 (источник баланса) — если баланс из 1C, скидки могут считаться в 1C. Если баланс custom — скидки в Parent App.

## Рассматриваемые варианты

### Вариант A — В источнике (1C или custom backend) — баланс приходит уже со скидкой
- **Описание:** Логика скидок применяется при формировании счёта в источнике (1C или наш backend). Parent App просто отображает финальную сумму.
- **Плюсы:**
  - Parent App простой — только отображение
  - Одна точка истины для скидок
- **Минусы:**
  - Льгота раннего платежа зависит от даты, на которую родитель открывает приложение. Источник должен пересчитывать баланс по запросу — иначе родитель увидит устаревшую сумму
  - Менять правила скидок в 1C сложнее, чем в Parent App
  - UX «вы могли бы заплатить меньше до 20 мая» сложнее реализовать

### Вариант B — В Parent App backend
- **Описание:** Источник присылает базовую сумму (без скидок). Parent App backend применяет правила: семейные скидки на основе семьи, временные на основе даты.
- **Плюсы:**
  - Полный контроль над UX скидок (видишь обратный отсчёт «до конца льготы 3 дня»)
  - Легко менять правила скидок
  - Расчёт всегда актуален на текущий момент
- **Минусы:**
  - Логика финансов разделена между источником (базовая ставка) и Parent App (скидки) — риск расхождения
  - При проверке бухгалтером истиной может стать «как насчитал Parent App» — нужно записывать применённые скидки в журнал платежа

### Вариант C (компромисс) — Семейные в источнике, временные в Parent App
- **Описание:** Семейная льгота применяется в источнике, потому что она привязана к составу семьи (стабильная характеристика, редко меняется). Льгота раннего платежа применяется в Parent App, потому что это UX-фича и меняется по времени.
- **Плюсы:**
  - Каждое правило живёт там, где оно естественно
  - UX-логика остаётся в UX
  - Финансовая стабильность семейной скидки сохраняется
- **Минусы:**
  - Две точки логики скидок — сложнее объяснить новому разработчику
  - При выверке нужно понимать, кто что применил

## Решение
**Вариант A — скидки считаются в 1C.** 1C — единая точка истины по финансам. Баланс приходит в Parent App уже с применёнными семейными льготами. Льготы раннего платежа в v1 не считаются (отложены в v1.1).

## Почему
- 1C владеет балансом (D-001) — естественно владеть и правилами его расчёта
- Бухгалтер управляет правилами скидок в привычной системе
- Меньше мест для бизнес-логики = меньше расхождений
- Меньше работы Алексу в v1 — Parent App только отображает

## Последствия
- В любом варианте: в журнале платежа должны записываться **исходная сумма, применённые скидки, итоговая сумма**. Это нужно для отчётности и спорных ситуаций.
- Если **Вариант B** или **C**: Parent App backend должен знать состав семьи (минимум — количество детей и их порядок зачисления).

## Влияние на интеграции с другими системами
- **Вариант A:** Изменения правил скидок требуют изменений в источнике (1C или backend). При замене Toddle / 1C нужно переносить правила.
- **Вариант B:** Правила в Parent App, что делает Parent App ответственным за бизнес-логику ценообразования. При смене источника правила переезжают вместе с приложением.
- **Вариант C:** Семейные скидки переезжают вместе с источником; временные остаются в Parent App.

## Зависимости
- D-001 (источник баланса) должен быть закрыт первым

## Пересмотреть когда
- Если правила скидок усложнятся (например, разные ставки для разных кампусов, программ, грейдов)
- При замене Toddle / 1C в 2027–28
