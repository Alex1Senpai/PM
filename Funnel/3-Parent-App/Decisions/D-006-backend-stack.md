# D-006 — Backend stack для Parent App

## Контекст
После D-001 (пересмотрено в Вариант C — Custom backend как кеш + 1C как источник истины), наш собственный backend становится центральным компонентом архитектуры Parent App. Он:

- Предоставляет REST/JSON API для Flutter-клиента
- Принимает webhook'и от Click и PayMe (платёжные подтверждения)
- Синхронизируется с 1C (модель синхронизации — D-007)
- Отправляет SMS через выбранного провайдера (T-002)
- Отправляет push-уведомления (FCM/APNs)
- Хранит данные пользователей (мульти-ролевая схема из D-004)
- Кеширует баланс и историю платежей
- В будущем — интеграции с Hikvision, Toddle, и т.д.

Решение о стеке backend критично, потому что:
- Влияет на скорость разработки Алекса и Тимура
- Определяет, насколько легко найти новых разработчиков в 2027+
- Задаёт операционную сложность (деплой, мониторинг, отладка)

Команда: Алекс (Flutter/Dart expert) + Тимур (intern, более ответственный). Дедлайн — 15 августа 2026 (12 недель).

## Рассматриваемые варианты

### Вариант A — Dart на сервере (Serverpod или эквивалент) (принят)
- **Описание:** Backend пишется на Dart, на одном из server-framework'ов (Serverpod — наиболее зрелый и Flutter-first; альтернативы Shelf, Conduit). База — PostgreSQL.
- **Плюсы:**
  - **Алекс уже Dart-эксперт.** Нулевая кривая обучения для основного разработчика.
  - **Один язык в стеке.** Flutter + backend оба на Dart. Меньше переключения контекста.
  - **Shared models.** Можно описать модели данных один раз, использовать и на клиенте, и на сервере. Серверная генерация кода клиента — Flutter всегда синхронизирован с API.
  - **Type safety end-to-end.** От базы данных до UI — одна система типов.
  - **PostgreSQL** — настоящий SQL, нужен для финансовых данных (транзакции, joins, сложные запросы).
  - Встроенные возможности (если Serverpod): auth, push, file storage, scheduling.
  - Для Тимура один язык учить вместо двух.
- **Минусы:**
  - Меньший ecosystem, чем у Node/Python — для редких интеграций придётся писать самим.
  - Меньшее community → меньше Stack Overflow ответов.
  - **Будущий найм:** Dart server-разработчиков в Ташкенте мало. Когда будем нанимать в 2027+, выбор кандидатов меньше.
  - Готовых библиотек для 1C нет — придётся писать REST-клиент к 1C самостоятельно (но это true в любом языке).
- **Стоимость:** Минимальная относительно команды. Будущая цена: сложность найма второго backend-разработчика.

### Вариант B — Node.js + TypeScript (NestJS / Fastify / Express)
- **Описание:** Backend на TypeScript. PostgreSQL или MongoDB. Любой популярный фреймворк.
- **Плюсы:**
  - Огромный ecosystem — библиотеки для всего.
  - Большой talent pool в Ташкенте → проще нанимать.
  - Стандартный выбор для современных мобильных backend'ов.
  - TypeScript = type safety.
- **Минусы:**
  - **Алекс не знает TypeScript** — нужно 1–2 недели на ramping. При 12-недельном дедлайне это потеря 10–15% времени.
  - Полиглот стек (Dart на клиенте, TypeScript на сервере) — больше когнитивной нагрузки.
  - Нет shared models — придётся синхронизировать схемы вручную или через OpenAPI/codegen.
- **Стоимость:** 2 недели ramping для Алекса. Будущая выгода: большой найм-пул.
- **Отклонён:** Потеря 2 недель ramping vs выгода найма позже — в текущем 12-недельном цикле это плохой обмен.

### Вариант C — Python + FastAPI
- Аналогично Node.js, но Python. Альтернатива — Django.
- **Минусы:** Алекс не знает Python. Та же проблема ramping.
- **Отклонён:** по той же причине, что и Node.js.

### Вариант D — Go
- **Минусы:** Крутая кривая обучения, ни Алекс, ни Тимур не знают, ecosystem слабее для мобильных backend'ов.
- **Отклонён:** не подходит.

### Вариант E — Firebase as Backend
- **Описание:** Firebase Auth, Firestore, Cloud Functions, FCM. Минимум backend-кода.
- **Минусы:**
  - **Firestore — NoSQL.** Финансовые данные требуют SQL (транзакции, joins, сложные агрегации). NoSQL — ошибка для нашего домена.
  - Vendor lock-in на Google.
  - Сложная business logic трудно реализуется в Firestore queries.
  - 1C-синхронизация всё равно потребует отдельный сервис.
- **Отклонён:** NoSQL для финансов — фундаментальная ошибка.

### Вариант F — Supabase + edge functions
- **Описание:** Open-source альтернатива Firebase. PostgreSQL underneath. Auth, storage, edge functions (Deno/TypeScript) включены.
- **Плюсы:** Меньше кода, real SQL, less vendor lock-in (можно перенести на свой Postgres).
- **Минусы:** Edge functions на TypeScript — добавляет второй язык. Supabase — относительно новая платформа.
- **Не выбран:** добавление второго языка нивелирует главное преимущество Варианта A.

## Решение
**Вариант A — Dart на сервере с PostgreSQL.**

Конкретный server-framework — выбор Алекса (Serverpod рекомендуется как наиболее зрелый Flutter-first вариант; альтернативы Shelf, Conduit). Алекс должен оформить этот выбор в отдельном Decision-файле (D-008) с обоснованием.

База данных: **PostgreSQL** (managed-инстанс — DigitalOcean Managed Postgres, AWS RDS, или эквивалент). NoSQL отклоняется однозначно — финансы требуют SQL.

## Почему
- **Velocity > ecosystem.** При 12-недельном дедлайне и команде из одного middle dev + интерна, скорость важнее размера ecosystem'а. Dart выигрывает по велосити с большим отрывом.
- **Shared models.** Один раз описать модели данных, использовать на клиенте и сервере. Огромный продуктивный множитель.
- **Один мыслительный контекст.** Алекс не переключается между Dart-mode и Node-mode. Тимур учит один язык, не два.
- **PostgreSQL.** Правильный выбор для финансовых данных. Никаких компромиссов с NoSQL.

## Принимаемые риски
- **Меньший ecosystem.** Для редких интеграций пишем своё. На v1 scope это управляемо.
- **Будущий найм.** В 2027+ нанять Dart server-разработчика в Ташкенте сложнее, чем Node/Python. Принимаем сознательно. Митигация: если рост требует найма — рассматриваем переписывание backend (но это будет с известной проблемой и временем на исследование).
- **Меньшая помощь сообщества.** Stack Overflow для Dart server — реже. Митигация: документация Serverpod хорошая, Flutter community активное.

## Последствия
- **Что мы теперь обязаны сделать:**
  - Алекс пишет D-008 — выбор конкретного server-framework на Dart (Serverpod / Shelf / Conduit / etc.) с обоснованием
  - Настроить инфраструктуру: managed PostgreSQL, сервер для Dart-приложения (DigitalOcean App Platform / AWS Lightsail / Render — Алекс выберет)
  - Настроить CI/CD для backend (тесты + автоматический деплой)
  - Спроектировать модель данных (User, Family, Child, Payment, etc.) с учётом мульти-роли (D-004) и shared с Flutter-клиентом
- **Что мы не можем легко поменять:**
  - Переписать backend на другой стек после v1 — несколько недель работы.
- **Что это открывает:**
  - Возможность переиспользовать backend для Teacher App и Admin Dashboard в будущем (если они тоже будут на Flutter)
  - Code sharing между всеми будущими Flutter-поверхностями

## Влияние на интеграции и LMS
- **Сейчас:** Backend подключается к 1C (D-007), Click, PayMe, SMS-провайдеру, FCM/APNs. REST/JSON во всех случаях — стек не имеет значения для контрактов.
- **2027–28 LMS:** Когда строим LMS — снова решаем. Возможные пути:
  - LMS пишется на том же Dart-стеке → переиспользование кода
  - LMS отдельный сервис на другом стеке → REST-интеграция между ними
- Решение не блокирует никакие варианты для будущего.
