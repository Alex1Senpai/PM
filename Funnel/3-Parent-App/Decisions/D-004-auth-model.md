# D-004 — Модель авторизации в Parent App

## Контекст
Родитель в Parent App получает доступ к финансовым данным (баланс, оплаты), данным о ребёнке (посещаемость, проходные события, переписка со школой). Авторизация должна:

1. Быть **достаточно надёжной для финансовых данных** — пара́метры, сравнимые с банковскими приложениями
2. Быть **знакомой родителю** — узбекская аудитория ожидает стандартный паттерн (Click, PayMe, банки)
3. Быть **доступной для всех родителей** — 100% покрытие, без зависимостей от Telegram, email, или внешних провайдеров идентичности
4. Иметь **минимальное трение** на повторном входе — родитель не должен ввести SMS-код каждый раз
5. **Поддерживать мульти-ролевую модель в будущем** — учителя, администраторы, ресепшен будут логиниться той же системой

## Рассматриваемые варианты

### Вариант A — Phone OTP через SMS + биометрия для повторного входа (принят)
- **Описание:** Родитель вводит номер телефона, получает SMS с кодом, входит. Сессия сохраняется на устройстве. Повторный вход — биометрический (Face ID / Touch ID). Раз в месяц или при подозрительной активности — повторная SMS-верификация.
- **Плюсы:**
  - 100% покрытие — у всех родителей есть телефон
  - Знакомый паттерн (Click, PayMe, банки)
  - Audit-friendly: «отправили OTP, родитель ввёл код»
  - Биометрия делает UX повторного входа невидимым
  - Не зависит от внешних аккаунтов (Telegram, Google, Toddle)
- **Минусы:**
  - SMS-доставка стоит денег и зависит от провайдера в Узбекистане
  - Если родитель меняет номер — нужна процедура восстановления
- **Стоимость:** ~$60/месяц на SMS при 1500 родителях × 2 авторизации/месяц × $0.02. Интеграция с SMS-провайдером — 1 неделя работы Алекса.

### Вариант B — Telegram OTP
- Отклонён. Telegram периодически блокируется в регионе, и стандарт «банковской» авторизации в Узбекистане — SMS, не Telegram. Для финансовых данных нестандартность создаёт ощущение «несерьёзного» приложения.

### Вариант C — Email + пароль
- Отклонён. Многие родители не проверяют email. Сброс пароля становится бесконечной службой поддержки.

### Вариант D — Google SSO
- Отклонён. Не все родители на Gmail. Создаёт зависимость от Google для аккаунта школы.

### Вариант E — SSO через Toddle
- Отклонён. Toddle планируется заменить в 2027–28. Привязывать авторизацию к системе, от которой мы уходим — плохая идея.

## Решение
**Вариант A — Phone OTP через SMS + биометрия для повторного входа. Схема пользователя готовится к мульти-роли.**

## Почему
- Покрытие всей аудитории
- Знакомый паттерн → меньше звонков в поддержку
- Audit-friendly для финансовых операций
- Не привязывает нас к Toddle, Google, или Telegram
- Биометрия закрывает UX-проблему повторного ввода SMS
- Стоимость SMS пренебрежимо мала по сравнению с доходом от обучения

## Архитектурные «семена» для мульти-роли (обязательны в v1)

Схема пользователя в backend Parent App должна с первого дня выглядеть так:

```
User:
  id (UUID)
  phone (verified, unique)
  role (enum: "parent" — в v1 единственное значение; в v1.1+ добавятся "teacher", "admin", "reception", "student")
  full_name
  created_at
  verified_at
  status (active / disabled)
```

В v1 на UI входа никакого выбора роли нет — все, кто логинятся, по умолчанию `parent`. Но поле `role` существует в БД и в API контракте. Когда добавляются другие роли в v1.1+, не нужно мигрировать схему.

**Авторизация (sessions) привязана к user, не к телефону.** Один user может иметь несколько активных устройств. При входе с нового устройства — обязательная SMS-проверка.

## Последствия
- **Что мы теперь обязаны сделать:**
  - Выбрать SMS-провайдера в Узбекистане (Eskiz, PlayMobile, или другой) — это discovery-задача Алекса, отдельный документ
  - Реализовать биометрический повторный вход через системные API (iOS LocalAuthentication, Android BiometricPrompt)
  - Настроить rate-limiting на endpoint OTP (защита от перебора SMS)
  - Создать процедуру восстановления при смене номера (через школу, не self-service)
- **Что мы не можем легко поменять:**
  - Привязка к телефону. Менять на email потом будет больно.
- **Что это открывает:**
  - Учителя, администраторы, ресепшен — все могут логиниться той же системой в v1.1+ без миграции схемы

## Влияние на интеграции и LMS
- Когда мы построим LMS в 2027–28: схема `User { phone, role, ... }` останется. LMS может либо использовать наш auth-сервис, либо иметь свой и обмениваться токенами. Контракт стабилен.
- Привязка к телефону НЕ зависит от Toddle — поэтому замена Toddle не требует пересмотра auth-модели.

## Зависимости
- Discovery по выбору SMS-провайдера (отдельная задача Алекса, до начала разработки)
- Решение о технологическом стеке клиента (D-005) — биометрические API различаются между нативными и кросс-платформенными движками

## Пересмотреть когда
- Если выбранный SMS-провайдер ненадёжен — рассмотреть второго (или Telegram как fallback)
- Когда в 2027–28 строится LMS — решить, использовать ли один и тот же auth-сервис
