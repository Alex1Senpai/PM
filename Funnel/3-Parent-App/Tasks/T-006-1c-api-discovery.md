# T-006 — Discovery: что предоставляет 1C школы для интеграции с Parent App backend

## Контекст
По D-001 + D-007 наш backend синхронизируется с 1C: push платежей в 1C + pull балансов из 1C. Это **не работает без того, чтобы 1C школы умел отдавать данные через какой-то API.**

1C — не один продукт. Это семейство конфигураций. То, что предоставляет конкретная установка 1C школы, мы **не знаем заранее.** Может быть прекрасный REST endpoint. Может быть только OData. Может быть выгрузка в Excel по расписанию. Может быть «приходи раз в неделю, скажи мне CSV».

Эта discovery-задача отвечает на вопрос: **«Какая интеграционная поверхность есть в нашем 1C сегодня, и что нужно построить (на стороне 1C), чтобы Parent App backend мог жить с ним?»**

Тимур делает её **в паре с 1C-программистом школы** (если есть) или с человеком, который ведёт 1C. Тимур не разработчик 1C — он только собирает факты. Конкретные решения по 1C-конфигурации делает 1C-программист.

**Спецификация:** [[../Specs/payments]]
**Связанные решения:**
- [[../Decisions/D-001-balance-source]] — 1C как источник истины
- [[../Decisions/D-007-1c-sync-model]] — гибридная модель синхронизации
**Стадия воронки:** 3-Parent-App

## Что нужно сделать

1. **Bekhruz сводит Тимура с 1C-программистом школы** (или с финдиректором, если программиста нет). Назначить 60-минутную встречу.

2. **На встрече собрать ответы по 8 вопросам:**
   1. Какая конкретно конфигурация 1C используется? (Например, «1C: Бухгалтерия 8.3», или кастомная конфигурация)
   2. Кто её сопровождает? (Внутренний человек, внешний подрядчик, фриланс?)
   3. Есть ли уже какой-то API наружу? (REST? OData? SOAP? HTTP-сервис?)
   4. Какие endpoints/функции уже опубликованы? Какие данные они отдают?
   5. Как авторизуется внешний клиент? (Token? Basic auth? IP whitelist?)
   6. Есть ли webhooks / push-механизм из 1C наружу при изменениях?
   7. Какова стандартная задержка ответа 1C на запросы? (Чтобы прикинуть polling-частоту)
   8. Готов ли 1C-сопровождающий **открыть новые endpoints**, если их нет? Каков срок и стоимость?

3. **Записать ответы** в раздел «Результат» этого файла. Формат — простой Q&A, не таблица.

4. **На основе ответов составить план для D-007:**
   - Если 1C уже отдаёт нужные данные — отметить какие, через какой механизм
   - Если 1C ничего не отдаёт — какие endpoints нужно построить на стороне 1C (список) и кто это делает
   - Какая частота polling реалистична с учётом latency 1C-API
   - Какой механизм push в 1C доступен для платежей (REST? Прямой insert в БД? Промежуточный stage table?)

5. **Эскалировать тревожные сигналы Bekhruz:**
   - Если 1C-сопровождающий говорит «это очень сложно, 2 месяца работы» → это **блокер для 15 августа**, нужно срочное решение (ускорять 1C-сторону, переходить на Вариант B из D-001, или менять scope)
   - Если 1C открыт только через «ручную выгрузку CSV» — нужен план переходный
   - Если стоимость работ 1C-стороны выше определённого порога (Bekhruz укажет) → ре-приоритезация
