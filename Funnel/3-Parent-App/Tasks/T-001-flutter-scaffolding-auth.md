# T-001 — Поднять Flutter-проект и реализовать экран авторизации (Phone OTP + биометрия)

## Контекст
Это первая буквальная задача по Parent App. Перед началом разработки любой бизнес-логики нужно:
1. Иметь работающий Flutter-проект с правильно настроенной структурой
2. Иметь экран входа, через который родитель попадает в приложение
3. Иметь схему пользователя в локальном хранилище, готовую к мульти-ролевой модели

После этой задачи Алекс сможет приступить к экрану баланса, к интеграции с 1C, к платежным провайдерам и так далее. Без этой задачи приложение не существует.

**Спецификация:** [[../Specs/payments]] (косвенно — payments требует auth)
**Решения, к которым задача относится:**
- [[../Decisions/D-004-auth-model]] — Phone OTP + биометрия + мульти-ролевая схема
- [[../Decisions/D-005-client-tech-stack]] — Flutter / Dart

**Стадия воронки:** 3-Parent-App

## Что нужно сделать
Каждый шаг — около 0.5–1 рабочего дня.

1. **Инициализировать Flutter-проект** в новом репозитории (название проекта — согласовать с Bekhruz перед коммитом, временно `oxbridge_parent_app`). Использовать Flutter 3.x stable. Минимум поддержки: iOS 15+, Android 9+.

2. **Настроить структуру проекта** по выбранной тобой архитектуре (Provider, Riverpod, Bloc — твоё решение, но обоснуй в Decisions/D-007-flutter-architecture.md). Не пытайся выбрать «самый модный» — выбери то, что тебе быстрее писать. В D-007 объясни, почему.

3. **Реализовать экран ввода номера телефона** — простой UI: поле для номера, кнопка «Получить код». Валидация: узбекский формат (+998), допустимая длина. Поддерживаемые языки: русский (v1 единственный).

4. **Реализовать экран ввода OTP-кода** — поле для 6-значного кода, авто-фокус, авто-сабмит при заполнении. Таймер «получить код повторно через N секунд». Кнопка «Изменить номер» возвращает на предыдущий экран.

5. **Реализовать сервис AuthService** — пока **моковый** (mock). Никакой реальной отправки SMS. AuthService должен:
   - Принимать номер телефона → возвращать moc'ed «код отправлен»
   - Принимать код → возвращать moc'ed User объект и токен сессии (просто строка)
   - Сохранять User и токен в локальном хранилище (secure storage)
   Когда T-002 (выбор SMS-провайдера) и backend будут готовы, AuthService переключим на реальную интеграцию.

6. **Реализовать схему локального хранения User**:
   ```dart
   class User {
     final String id;
     final String phone;
     final String role; // "parent" в v1
     final String fullName;
     final DateTime createdAt;
     final DateTime verifiedAt;
   }
   ```
   Поле `role` обязательно — даже если в v1 всегда `"parent"`. Это «семена» для мульти-роли (см. D-004).

7. **Реализовать биометрический повторный вход**:
   - При запуске приложения, если есть сохранённый токен → запрос биометрии (Face ID / Touch ID на iOS, BiometricPrompt на Android) → при успехе пускаем в приложение
   - Пакет — `local_auth`
   - При неуспехе биометрии → пускаем на экран ввода OTP

8. **Реализовать экран «Главная» (заглушка)** — пустой экран с текстом «Привет, {имя родителя}». Это просто проверка, что после авторизации пользователь куда-то попадает. Реальный контент главного экрана — отдельные задачи.

9. **Настроить навигацию** — Splash → (OTP flow или биометрия) → Home. Используй стандартный Flutter Navigator или go_router.

10. **Базовый CI** — настроить GitHub Actions для билда iOS и Android при пуше в main и в PR. Без публикации в магазины — только сборка.

## План тестирования
- **Ручные проверки:**
  - Ввести номер, получить «код» (моковый), ввести код, попасть в Home
  - Закрыть приложение, открыть снова → должна запроситься биометрия
  - Удалить и переустановить → начало с экрана ввода номера
- **Автотесты:**
  - Unit-тесты на AuthService (моковая отправка возвращает корректную структуру)
  - Widget-тесты на экран OTP (ввод 6 цифр триггерит auto-submit)
- **Граничные случаи:**
  - Неверный формат номера → валидация показывает ошибку
  - Истекший таймер OTP → кнопка «получить заново» активна
  - Биометрия отключена в настройках устройства → fallback на OTP
  - Закрытие приложения во время OTP → состояние не теряется (повторный экран ввода кода доступен)
