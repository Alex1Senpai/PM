# Статистика: пятничный дашборд команды продаж (T12)

## Что это
Один экран со сводной статистикой работы команды приёма по данным amoCRM: сквозная конверсия «заявка → зачислен» в процентах, гигиена работы с лидами (первый контакт, задачи, просрочка), одна строка на менеджера, сравнение кампусов МЮ/ЯШ с разбивкой по источнику и программе, причины закрытия и медианное время лида в каждой стадии воронки. Открывается из amoCRM в один клик. Служит руководителю для пятничного разбора.

## Почему сейчас
Пятничный разбор сегодня собирается руками из amoCRM-выгрузок. Уже видны «кладбища» сделок («Лист ожидания» — 29 дней без касания, «Договор отправлен» — медианный возраст 61 день), но систематического детектора нет. Без общей цифры недели команда управляет ощущениями, а не процентами.

## Пользователи и их задача
- **Основной пользователь:** руководитель отдела приёма (и Bekhruz).
- **Задача:** в пятницу открыть один отчёт и за 10 минут увидеть: общую конверсию недели и динамику, кто из 8 менеджеров выбивается, какой кампус/источник проседает, где умирают сделки.
- **Где находится:** десктоп, внутри amoCRM.
- **Частота:** еженедельно (пятница) + точечные заглядывания в течение недели.

## Архитектура (зафиксированное решение)
**Бэкенд-агрегатор + тонкий виджет.** Расчёт метрик живёт в существующем сервере `remix-of-oxbridge` (там уже есть рабочий OAuth-клиент amoCRM и Postgres/Neon) как изолированный модуль `stats`. Виджет amoCRM — отдельный репозиторий `amocrm-widget`, чистый рендер готового JSON.

```
amoCRM v4 ──(крон ночью + on-demand refresh)──▶ extractors ──▶ compute ──▶ Postgres(Neon)
  leads · tasks · events · users · pipelines · loss_reasons        │ stats_cache + stats_weekly_snapshot
                                                                    ▼
              GET /api/v1/stats/report ──JSON──▶ виджет amoCRM (тонкий рендер)
```

- **Быстро:** виджет читает посчитанный отчёт из `stats_cache` — открытие без ожидания amoCRM API.
- **Динамично:** кнопка «⟳ обновить» форсирует пересчёт «лёгких» метрик (задачи, лиды без задач, доли) в реальном времени; TTL кэша 15 минут.
- **Неделя-к-неделе:** ночной крон пишет недельные снимки в `stats_weekly_snapshot`.
- **Тяжёлое (медианы по стадиям из events API, лимит 7 rps):** считает только ночной крон; live-refresh переиспользует последнее ночное значение.

## Определения метрик (бизнес-правила)
Все проценты считаются в таймзоне **Asia/Tashkent (UTC+5)**, неделя — **Пн–Вс (ISO)**.

| # | Метрика | Формула / источник |
|---|---|---|
| 1 | Сквозная конверсия заявка→зачислен, % | **Когортная**: лиды, созданные в отчётную неделю; % достигших стадии «зачислен» к моменту расчёта. WoW = та же формула за предыдущую неделю. Свежие когорты занижены (лиды не дозрели) — отчёт помечает это. |
| 2 | % первого контакта в тот же день | Первая исходящая активность по лиду (events API: исходящий звонок / исходящее сообщение / закрытая задача / примечание — список типов конфигурируем) в тот же календарный день, что и создание лида. |
| 3 | % лидов без открытой задачи | Активные лиды (не в финальных статусах 142/143) с пустым `closest_task_at` / все активные. Цель 0%. |
| 4 | % просроченных задач | Незакрытые задачи с `complete_till < now` / все незакрытые задачи. |
| 5 | Доля закреплённых лидов на менеджера | Активные лиды менеджера / все активные. Норма при 8 менеджерах ~12,5%. |
| 6 | Личная сквозная конверсия | Метрика #1 с фильтром по `responsible_user_id`. Без поэтапной разбивки. |
| 7 | % без задачи и % просрочки на менеджера | Метрики #3 и #4 в разрезе `responsible_user_id`. |
| 8 | Конверсия по кампусу + разбивки | Кампус **по воронке** («Конверсия МУ» 10482294 → MU, «Конверсия ЯШ» 10892314 → YA; env `STATS_MU_PIPELINE_IDS`/`STATS_YA_PIPELINE_IDS` или автоопределение по имени); поле лида 996203 — fallback. Разбивка конверсии по источнику (поле «Источник») и программе (поле 991841 «Возраст детей»). |
| 9 | Причины закрытия, % | Распределение `loss_reason_id` (справочник `/api/v4/leads/loss_reasons`) среди лидов, закрытых как «не реализовано» (статус 143) за период. |
| 10 | Медиана дней в стадии (velocity) | События `lead_status_changed` за окно 90 дней → длительности пребывания в каждой стадии → медиана по стадии. По воронке, не по менеджеру. Подсветка стадий-«кладбищ». |

**«Зачислен»** = попадание лида в конфигурируемый статус: env `STATS_ENROLLED_STATUS_IDS` (список id через запятую). Если env не задан — автообнаружение по имени статуса (регэксп `/зачисл|первый платёж/i` по справочнику pipelines) с записью предупреждения в лог. Охват воронок — env `STATS_PIPELINE_IDS`; не задан — все воронки, кроме `career`.

## Что функция ОБЯЗАНА делать (функциональные требования)
1. **БД**: миграция `0007_stats_dashboard.sql` создаёт `stats_weekly_snapshot(week_start date, scope text, key text, payload jsonb, computed_at timestamptz, PK(week_start, scope, key))` и `stats_cache(id int PK, report jsonb, heavy jsonb, computed_at timestamptz)` — `heavy` хранит артефакты ночного пересчёта (velocity-медианы + enrolled-IDs из events), которые переиспользует light-refresh. Идемпотентно (`create table if not exists`), в стиле существующих миграций.
2. **Экстракторы amoCRM** (`server/src/services/stats/`): leads (с custom fields, пагинация 250), tasks, users, pipelines+statuses, loss_reasons, events (`lead_status_changed` и типы «первого контакта», пагинация 100) — все через существующий `amoCRMService.getValidAccessToken()`, с общим троттлером ≤7 запросов/сек и ретраем на 429.
3. **Расчёт** (`statsCompute.ts`): чистые функции «(сырые сущности) → отчёт» по правилам из таблицы выше. Без I/O — тестируются на фикстурах.
4. **API**:
   - `GET /api/v1/stats/report?week=YYYY-Www` — отдаёт отчёт из кэша/снимков; без `week` — текущая неделя. Защита: Bearer-токен `STATS_API_TOKEN` (сравнение через `safeTokenEquals`), rate-limit как у admin-роутов.
   - `POST /api/v1/stats/refresh` — форсирует пересчёт лёгких метрик (лиды/задачи/доли), переиспользует последние ночные медианы; обновляет `stats_cache`; тот же Bearer.
   - `GET /api/v1/stats/cron` — ночной полный пересчёт (включая events/медианы) + upsert снимка текущей недели. Защита: `requireOpsToken` (CRON_SECRET), как у `/ops/monitor`.
5. **Крон**: запись в `vercel.json`: `{"path": "/api/v1/stats/cron", "schedule": "0 22 * * *"}` (22:00 UTC = 03:00 Ташкент).
6. **Виджеты** (`amocrm-widget/`): **шесть независимых интеграций** amoCRM (решение Bekhruz 2026-06-11: статистика по кампусам — отдельными виджетами), общее рендер-ядро `src/core.js`, сборка → 6 zip: `oxbridge_stats_team` (KPI-карточки команды), `oxbridge_stats_managers`, `oxbridge_stats_campus_mu` / `_campus_ya` (конверсия кампуса + WoW + источники/программы/причины), `oxbridge_stats_velocity_mu` / `_velocity_ya` (время в стадии воронки кампуса). Каждый: area `advanced_settings`, AMD `script.js`, i18n ru/en, настройки: базовый URL API + токен; навигация по неделям ◀ ▶, «⟳ обновить», ⚠-маркеры (без задачи >5%, просрочка >15%, доля лидов вне 8–17%, медиана стадии >14 дней).
7. **Деградация**: если снимка прошлой недели нет (первый запуск) — WoW показывает «—», не ошибку. Если events недоступны — блок velocity показывает дату последнего успешного расчёта.
8. **Сборка виджетов**: `npm run build` генерирует из общего ядра 6 архивов `dist/oxbridge_stats_*.zip` — каждый загружается в amoCRM как отдельная интеграция.

## Что функция ДЕЛАТЬ НЕ ДОЛЖНА (вне scope)
- Поэтапная воронка по каждому менеджеру (смотрим вручную при выбросе).
- Запись чего-либо в amoCRM (виджет и бэкенд read-only по отношению к CRM).
- Realtime-обновления (websockets), пуши, рассылка отчёта.
- Исторический бэкфилл глубже, чем позволяет events API (90 дней для velocity достаточно).
- Дашборд вне amoCRM (отдельный сайт) — только виджет.
- Автораспределение лидов, алерты менеджерам.

## Критерии приёмки
- [ ] Руководитель открывает виджет в amoCRM → видит все 10 метрик одним экраном за < 2 сек (из кэша).
- [ ] Сквозная конверсия недели сходится с ручной проверкой по 2–3 лидам (DoD из ТЗ).
- [ ] Метрики #3, #4 после «⟳ обновить» отражают текущее состояние задач в amoCRM (проверка: создать/закрыть задачу → refresh → цифра изменилась).
- [ ] WoW-динамика появляется со второй недели работы; до этого «—».
- [ ] Все расчётные функции покрыты юнит-тестами на фикстурах; `npm exec nx run server:test` зелёный.
- [ ] Эндпоинты без валидного токена отвечают 401/403; крон-эндпоинт без CRON_SECRET в проде — 503/401.
- [ ] Троттлер не превышает 7 rps к amoCRM (юнит-тест на таймлайн вызовов).

## Зависимости
- **Внутренние:** рабочие OAuth-токены amoCRM в Neon (`AmoCRMOAuthToken`), `getPool()`, `requireOpsToken`, `safeTokenEquals` — всё уже в `remix-of-oxbridge`.
- **Внешние:** amoCRM API v4 (`/leads`, `/tasks`, `/users`, `/leads/pipelines`, `/leads/loss_reasons`, `/events`), права интеграции на чтение этих сущностей.
- **Данные/конфиг от Bekhruz при деплое:** реальные `STATS_ENROLLED_STATUS_IDS`, `STATS_PIPELINE_IDS`, `STATS_API_TOKEN` (новый секрет), список типов «первого контакта», если дефолт не подходит.

## Нефункциональные требования
- **Производительность:** отчёт из кэша < 2 сек; ночной полный пересчёт укладывается в лимит Vercel-функции (троттлинг + окно 90 дней; при риске таймаута — события постранично с продолжением в следующем запуске крона).
- **Доступность:** некритичный сервис; при сбое amoCRM показываем последний успешный расчёт с меткой времени.
- **Безопасность:** PII (имена/телефоны лидов) в отчёт не попадают — только агрегаты и имена менеджеров; токены — в env; эндпоинты закрыты Bearer-токенами.
- **Языки:** интерфейс виджета ru (основной), en (заглушка-перевод).
- **Устройства:** десктоп внутри amoCRM.
