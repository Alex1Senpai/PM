# Как правильно работать через GitHub — рабочий процесс для PR

Тимур, прочти один раз внимательно. Дальше держи под рукой как чек-лист. Этот документ — про **процесс**, а не про задачу. Что именно делать в задаче — всегда написано в её `T-NNN-*.md` файле.

Цель: каждый твой PR доходит до merge с первой попытки, без переделок и без вопросов «а ты точно это сделал?».

---

## Главный принцип

**PR = одна задача = одна ветка = один merge.**

Если ты ловишь себя на том, что в одном PR смешалось два разных дела (например, и UTM-теги, и фикс типографики) — останови, разнеси на два PR.

---

## Шаг 1. Перед началом работы

1. Открой задачу в `Funnel/<воронка>/Tasks/T-NNN-*.md`. Если задачи нет — **стоп**. Напиши Bekhruz, попроси создать. Работа без задачи = работа без понятного «готово».

2. В этом файле, в верхнем блоке (frontmatter, между двумя `---`), измени:
   - `status: pending` → `status: in_progress`
   - `assignee: Unassigned` → `assignee: Timur` (если ещё не стоит)
   - `commit_initial: ""` — оставь пока пустым, заполнишь после первого коммита

3. Сохрани этот маленький правочный коммит. Сообщение коммита:
   ```
   T-NNN: start work
   ```

   Это говорит Bekhruz, что задача в работе.

---

## Шаг 2. Создать ветку **правильно**

**Никогда не работай на `patch-1`, `patch-2`, `patch-3`.** Это автоматические имена, которые GitHub UI создаёт по умолчанию — они ничего не говорят о задаче и путают историю.

**Правильное имя ветки:**
```
task/T-NNN-короткое-описание
```

Примеры:
- `task/T-002-sms-discovery`
- `task/T-005-ads-utm-sprint`
- `task/T-009-toddle-api`

Только английские буквы, цифры и дефисы. Никаких пробелов, никакой кириллицы в имени ветки.

### Как создать ветку через GitHub UI

1. Открой репозиторий PM на GitHub.
2. Слева сверху найди дроп-даун с надписью `main`.
3. В поле «Find or create a branch...» начни печатать имя: `task/T-002-sms-discovery`.
4. GitHub предложит «Create branch: task/T-002-sms-discovery from main» — нажми.

Готово. Дальше все правки делай **только в этой ветке**. После merge ветку удалишь (см. Шаг 8).

---

## Шаг 3. Делай работу — маленькими коммитами

Один большой коммит «сделал всё» — плохо. Bekhruz не сможет посмотреть твою логику. Лучше 4–5 маленьких коммитов, каждый про одну вещь.

**Формат сообщения коммита:**
```
T-NNN: <глагол в повелительном> <что сделал>
```

Примеры хороших коммитов:
- `T-002: добавить таблицу провайдеров`
- `T-002: написать рекомендацию по Eskiz`
- `T-002: заполнить раздел "Открытые вопросы"`

Примеры **плохих** коммитов (так не делай):
- `update`
- `fixes`
- `WIP`
- `работа`
- `сделал` (что именно?)

Префикс `T-NNN:` обязателен. Без него Bekhruz не сможет связать твой коммит с задачей.

---

## Шаг 4. Открыть Pull Request

Когда работа готова — открой PR. **Не раньше.** Не открывай PR, пока в нём пусто или половина сделана. Если хочешь показать промежуточный результат — пиши в Telegram.

### Как открыть PR через GitHub UI

1. Зайди в свою ветку через GitHub UI.
2. Сверху появится жёлтая полоска с кнопкой «Compare & pull request» — нажми.
3. Проверь: **base = `thebekhruz/PM` `main`**, **compare = твоя ветка**.

### Название PR

Формат:
```
T-NNN: <одна фраза в повелительном тоне>
```

Примеры:
- `T-002: SMS provider discovery — рекомендация Eskiz`
- `T-005: завершение weekend marketing sprint`
- `T-009: discovery Toddle API`

### Описание PR — обязательный шаблон

Скопируй этот шаблон и заполни. **Каждый раз. Без исключений.**

```markdown
## Задача
Closes: `Funnel/<воронка>/Tasks/T-NNN-<slug>.md`

## Что сделал
Один абзац своими словами. Не повторяй задачу — расскажи, **что именно** ты сделал.

## Как Bekhruz проверит
Перечисли 2–3 конкретных шага:
- Какие разделы файла теперь заполнены
- Где смотреть результат (если есть таблица, ссылка, скриншот)

## Чек-лист "Готово" (Definition of Done)
Скопируй чек-лист из задачи. **Поставь галочки ТОЛЬКО на то, что реально сделал.**

- [ ] Пункт 1
- [ ] Пункт 2
- [ ] Пункт 3

## Что не получилось / открытые вопросы
Будь честен. Если что-то не сделал — напиши это здесь. Не оставляй просто галочку.

## Вне scope
Что я НЕ делал в этом PR, хотя могло показаться, что должен был. Защищает от вопросов в ревью.
```

---

## Шаг 5. ВАЖНО — Definition of Done честно

Это самая важная часть всего документа. Прочти дважды.

**Чек-лист в задаче — это контракт. Если ты ставишь галочку, значит ты это сделал.** Не «начал», не «попытался», не «частично» — **сделал**.

### Как должно быть

Если в задаче сказано «Тестовые SMS отправлены, время доставки записано» — это значит:
- Ты реально получил тестовый доступ к провайдеру
- Ты реально отправил пять-десять SMS на свой телефон
- Ты записал в файл, сколько секунд шло каждое SMS
- Это видно в файле как реальные данные

Если ты не смог получить тестовый доступ — **галочка остаётся пустой**, а в разделе «Открытые вопросы» ты пишешь, почему. Это **нормально**. Это **честно**. Bekhruz сможет помочь — например, дать тебе корпоративные данные школы для регистрации.

### Как **нельзя** делать

❌ Ставить `[x]` на пункт «Тестовые SMS отправлены», а через 5 строк в «Открытых вопросах» писать «тестовый доступ получить не удалось». Это противоречит само себе. Bekhruz это **сразу** увидит, и в следующий раз он не поверит ни одной твоей галочке. Доверие сложно вернуть.

❌ Ставить галочки «оптом» в конце, не глядя.

❌ Ставить галочку, чтобы «выглядело красиво».

### Правило

**Лучше PR с тремя честными галочками и четырьмя пустыми, чем PR с семью галочками, половина из которых липовые.**

Bekhruz больше ценит честность, чем зелёный чек-лист.

## Шаг 6. Проверить frontmatter перед отправкой PR

Frontmatter — это верхний блок между `---` и `---`. Он легко ломается, если не аккуратно.

**Что проверить:**

1. **Только два `---`:** один в начале, один в конце. Не три, не четыре.

   Правильно:
   ```yaml
   ---
   type: task
   id: T-002
   status: done
   ---

   # T-002 — ...
   ```

   **Неправильно** (так было в твоём PR #3):
   ```yaml
   ---
   type: task
   ...
   ---
   ---       ← лишний разделитель, всё ломается
   ```

2. **Значения с дефисами, дробями, спецсимволами — в кавычках.** Все остальные можно без.

   Пример:
   ```yaml
   status: done                          # без кавычек ок
   funnel-stage: "3-Parent-App"          # с дефисом → в кавычки
   code_repo: "n/a — discovery"          # em-dash → в кавычки
   ```

3. **Поле `status`** — только одно из трёх значений: `pending`, `in_progress`, `done`. Никаких других.

4. **После завершения задачи:**
   - `status: done`
   - `commit_final: "<sha последнего коммита перед merge>"`
   - Раздел «Завершение» внизу файла заполнен (один-два абзаца, что сделал)

## Шаг 7. Ревью

Bekhruz придёт в PR и оставит комментарии. Возможные варианты:

- **«Approve»** — можно мержить, переходи к Шагу 8.
- **«Request changes»** — нужно что-то поправить. Делай правки в той же ветке (новые коммиты), они автоматически появятся в PR. Не открывай новый PR.
- **Вопросы в комментариях** — отвечай **в самом PR**, не в Telegram. Это нужно, чтобы решения остались в истории.

### Что делать, если Bekhruz нашёл ошибку

1. Не оправдывайся. Просто исправь и ответь «Поправил».
2. Если не согласен — спокойно объясни почему. Bekhruz слушает.
3. Если не понял замечание — переспроси. Лучше задать вопрос, чем сделать неправильно.

---

## Шаг 8. После Merge — обязательные действия

После того, как Bekhruz нажмёт «Merge»:

1. **Удали ветку.** GitHub предложит кнопку «Delete branch» сразу после merge — нажми. Если забыл — зайди в `Branches`, найди свою ветку, удали.

2. **Обнови файл задачи.** Зайди на `main`, открой свой `T-NNN-*.md`, проверь:
   - `status: done` ✓
   - `commit_final` заполнен SHA финального коммита ✓
   - Раздел «Завершение» написан (если ещё не написал — допиши и сделай маленький коммит прямо в `main` под названием `T-NNN: fill завершение`)

3. **Напиши Bekhruz в Telegram:** «T-NNN готово, можно брать следующее». Одна строка.

---

## Чего **никогда** не делать

- ❌ Открывать PR из ветки `patch-1` / `patch-2` (это значит ты не создал нормальную ветку)
- ❌ Использовать одну и ту же ветку для двух разных задач
- ❌ Делать коммиты без префикса `T-NNN:`
- ❌ Ставить галочку на DoD-пункт, который ты не выполнил
- ❌ Закрывать (Close) свой PR без объяснения — лучше написать комментарий «Закрываю в пользу PR #X, потому что Y»
- ❌ Мержить свой PR сам, без approval от Bekhruz
- ❌ Редактировать чужие задачи в том же PR — только свою

## Чек-лист перед нажатием «Create Pull Request»

Пройди по нему **каждый раз** перед открытием PR. 30 секунд.

- [ ] Ветка называется `task/T-NNN-короткое-описание`, **не** `patch-N`
- [ ] Все коммиты начинаются с `T-NNN:`
- [ ] Frontmatter задачи: `status: done`, нет лишних `---`, все спецсимволы в кавычках
- [ ] Раздел «Результат» в задаче заполнен реальными данными
- [ ] DoD-чек-лист отражает реальность — не больше галочек, чем сделано
- [ ] Раздел «Завершение» в задаче написан (1–2 абзаца)
- [ ] В описании PR использован шаблон выше, секции «Что сделал» и «Открытые вопросы» заполнены
- [ ] Название PR: `T-NNN: одна фраза`

Если хотя бы один пункт не выполнен — не открывай PR. Поправь, потом открой.

## Шпаргалка по типичным ситуациям

### «Я начал работу, но завис на 2 дня — что делать?»
1. Не паникуй.
2. Напиши Bekhruz в Telegram: «Завис на T-NNN, потому что <X>. Что делать?»
3. Не оставляй задачу в `in_progress` молча больше двух дней. Лучше открытый блокер, чем тихая прокрастинация.

### «Я сделал работу, но потом понял, что часть неправильная»
1. Не удаляй PR.
2. Сделай дополнительные коммиты с исправлениями в той же ветке.
3. Опиши, что изменил, в комментарии к PR.

### «Я ошибся — открыл PR не из той ветки / коммит улетел не туда»
1. Скажи Bekhruz **сразу**, не пытайся скрыть.
2. Bekhruz поможет: либо переделать PR, либо игнорировать и открыть правильный.
3. Молчать про ошибку — самое худшее, что можно сделать.

### «Bekhruz попросил поправить что-то, чего я не понимаю»
Переспроси в PR-комментарии. Пример: «Уточни, пожалуйста, что именно надо изменить в этом разделе?»

### «У меня PR висит уже три дня без ответа»
Напиши Bekhruz в Telegram: «PR #N висит — посмотришь?» Один раз. Не каждый день.

## Резюме одним абзацем

Создаёшь ветку `task/T-NNN-slug`. Делаешь маленькие коммиты с префиксом `T-NNN:`. Когда закончил **всю** задачу — открываешь PR с шаблоном, в котором честный чек-лист (только реально сделанное). Bekhruz ревьюит. Ты правишь, если нужно. После merge — удаляешь ветку, обновляешь раздел «Завершение». Пишешь «готово» в Telegram.

Это весь процесс. Если делаешь это аккуратно — твои PR будут мержиться с первого раза, и Bekhruz будет давать тебе более интересные задачи.

---

## Если что-то непонятно

Спрашивай Bekhruz в Telegram **до того**, как сделаешь, а не после. Один вопрос экономит часы переделок.
